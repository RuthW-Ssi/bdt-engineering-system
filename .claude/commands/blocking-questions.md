# /blocking-questions — Cross-source blocker review

Aggregates blockers from both the wiki (durable) and Notion (live, via snapshot)
into one markdown report. Useful for standup, sprint review, or planning meetings.

Read-only — does not modify any source.

---

## Why snapshot-first

Notion's `query-database-view` MCP tool requires a Business plan, and `notion-search`
inside a data source does semantic search (not property filtering). So directly
filtering `Status = Blocked` over MCP is not reliable on the free/Plus tier.

Solution: rely on the markdown snapshot generated by `/sync-sprint`. The snapshot
encodes Blocked tasks/features with the `[!]` marker, so this command becomes a
pure file-grep + cross-reference operation.

---

## Steps

### 1. Read wiki blockers (Obsidian)

- Read `/Users/michel-angelo/Documents/bdt/knowledge-base/projects/bdt-engineering-system/wiki/_open-questions.md`
- Read `/Users/michel-angelo/Documents/bdt/knowledge-base/projects/bdt-engineering-system/wiki/_contradictions.md`
- Parse each section into tuple `(id, title, owner, blocks, source)`

### 2. Read Notion blockers (snapshot-first)

Look in `/Users/michel-angelo/Documents/bdt/knowledge-base/projects/bdt-engineering-system/pm/_snapshots/`:

**2a. Snapshot present and fresh (≤24h old per header timestamp)**
- Glob `pm/_snapshots/sprint-*.md`
- Parse the `_Mirror of Notion <id> as of <ISO-ts>_` header
- Within each snapshot, find:
  - Lines starting with `### ` (feature) where the next line indicates `Status: Blocked` or feature heading contains `(Blocked,`
  - Lines starting with `- [!] ` (Blocked task marker per `/sync-sprint` emoji convention)
- Extract: feature name + Blocked By context · task name + parent feature · any Wiki Refs URLs

**2b. Snapshot missing OR stale (>24h)**
- Print warning: `⚠ snapshot missing or stale — running live fallback (slower, partial)`
- Live fallback: read agent memory `notion-bridge.md` for DB IDs. Run
  `notion-search "Blocked"` with `data_source_url` set per DB to surface candidate
  rows. Acknowledge in the report that live results may be incomplete.
- After live fallback, prompt user: "Run `/sync-sprint` to refresh the snapshot
  before next call."

### 3. Render report (4 sections)

```
# Blocking Items — <ISO-ts>
_Snapshot age: <delta from header>_  (or `_Live fallback (snapshot stale/missing)_`)

## 🟡 Wiki — Open Questions (N unresolved)
- **q1** — <title> · Owner: <owner> · Blocks: <blocks> · [Open in Obsidian](obsidian://...)
- **q2** — ...

## 🔴 Wiki — Contradictions (N contested)
- **<slug>** — <topic> · Sources: <a> vs <b>
- ...

## 🟠 Notion — Blocked Features (N)
- **<feature name>** (Sprint <n>) — Blocked By: <blocked-by-text from snapshot> · [Open in Notion](<url-if-present-in-snapshot>)
- ...

## ⚫ Notion — Blocked Tasks (N)
- **<task name>** (Feature: <feature name>) · [Open in Notion](<url-if-present>)
- ...

## Summary
Total blockers: N (wiki: K, Notion: M)
Snapshot freshness: <e.g. "fresh, 2h old" / "stale 36h" / "missing">
Cross-references found: <list of wiki q-ids that map to Notion blocked items, marked ✓>
```

### 4. Highlight cross-references

For each Notion Blocked item, scan its `Blocked By` field (or surrounding context
in the snapshot) for tokens like `q1`, `q2`, etc. If matched against
`wiki/_open-questions.md` entries, append `✓ matched` to that line in the report.
This validates the bidirectional bridge is intact.

If a wiki q-id has no matching Notion entry, note in Summary as
`unmatched wiki questions: q5, q6, ...` (informational, not an error — some
questions are forward-looking or out-of-sprint scope).

---

## Output

Print to stdout. Do not write to disk.

---

## Anti-patterns

- ❌ Calling `notion-query-database-view` (requires Business plan; fails on most workspaces)
- ❌ Modifying wiki or Notion during the scan (read-only)
- ❌ Listing items that aren't blocked (filter strictly on `[!]` marker / `Status: Blocked`)
- ❌ Skipping the cross-reference check (key value of this command)
- ❌ Running live fallback silently — always declare snapshot freshness in output
